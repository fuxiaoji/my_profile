人在工学椅，刚把腿搭在桌子上。
在这个问题下，我想分享一个纯粹被“懒惰”逼出来的开源项目。
事情的起因，是在圈子里火得一塌糊涂的“Vibe Coding”概念。现在网上随便一搜，到处都是大佬们在晒自己的极客桌面：左边一块竖屏写代码，右边一块带鱼屏看文档，桌上摆着几千块钱的分体式人体工学键盘。敲起代码来不仅劈啪作响，还得配上屏幕挂灯、RGB 氛围灯和 Lofi Hip-hop 音乐。动不动就跟你聊“AI 赋能”、“沉浸式心流”。
前阵子我还在 B 站刷到了这种专门为 AI 编程定制的物理外设，vibecoding 键盘
几个定制的机械轴体和旋钮看着视频我直呼好家伙。
但当我激情下单前，看着自己花大几千买的人体工学椅，突然意识到一个极其破灭的现实：
无论你的键盘多酷炫、轴体多润滑、分体角度多符合人体工学……只要用它们，你就必须老老实实地“坐”在桌子前。
所谓人体工学的终极奥义，难道不是“躺平”吗？当你整个人向后死死瘫在椅子上，双脚搭在桌沿，甚至冬天想把双手死死裹在毛毯里的时候，你只要一伸手去够桌面上那个帅气的 Vibe 键盘，手腕悬空，肩膀发力——什么沉浸式心流，什么人体工学，瞬间全破功了。
直到那个周末，我正四仰八叉地躺在椅子上，手里握着手柄百无聊赖地刷着 B 站（switch 的 b 站插件）。手柄的握把完美贴合掌心，双手不需要被固定在桌面上，可以自然垂落在身体两侧，大拇指微动就能掌控一切。
那一刻我悟了：手柄，才是人类工业设计史上最伟大的人体工学设备！ 凭什么这种神仙外设只能用来打游戏？现在的 AI 编程早就不需要我们疯狂敲击键盘了，我们更多是在“审查”代码。既然是审查，用手柄难道不是绝杀？
去网上搜了一圈，现有的映射软件（比如 AntiMicroX）配置繁琐，且只能做最死板的键鼠映射，完全跟不上现代的 AI 工作流。既然没现成的，那干脆自己搓一个。
于是就有了这个项目：vibepad（手柄操控电脑 / Vibe Coding 快捷映射）。

它能把你的 Xbox、PS5、Switch Pro 等任何手柄，变成一个全能的控制器取代鼠标键盘。
我给它设计了三种核心模式，平时只需要按手柄的 SELECT 键就能无缝循环切换：
1. 纯正的“赛博街溜子”模式（Mouse Mode）
这是最基础的鼠标模拟，但我针对“瘫着上网”做了极度优化。
左摇杆就是鼠标，右摇杆就是滚轮（支持上下左右滑）。A 键左键，B 键右键。为了让你刷 B 站或 YouTube 时爽到极致，我把十字键直接映射成了媒体控制（上下调音量，左右快进快退）。
最核心的一个细节是扳机加速：用摇杆控制光标其实很慢，所以我做了一个机制，只要按住 RT 扳机键，光标移速会瞬间飙升 3.4 倍。这意味着你可以慢速精准点掉网页上的小广告，而遇到需要大跨度跨屏的时候，一扣扳机，光标直接“唰”地飞过去，指哪打哪。
2. 皇帝批阅奏折模式（Vibe Coding Mode）
既然有了 GitHub Copilot 和 Claude Code，咱们早就从“打字员”晋升为“代码审查员”了。在鼠标模式下，只要你长按 LB 键（类似键盘的 Shift 切层），手柄就会瞬间变成 AI 编程专用的控制器：

3. 野心爆棚但也最折磨人的模式（Vim 空间导航）
用摇杆模拟鼠标指针有个天生的缺陷：就算有加速，用摇杆去找像素点还是很累。
我就想，为什么 Switch 和 Apple TV 的界面用手柄操作就那么顺滑？因为它们是空间焦点导航！十字键一按，焦点自动吸附到下一个按钮上，根本不需要光标。
于是我利用 Windows 的 UI Automation（UIA）API，硬搓了一个 Vim 模式：
开启后，程序会扫描你屏幕上所有的可交互元素（按钮、超链接、输入框等），然后用一个透明的 Win32 置顶窗口在当前焦点上画一个蓝色的光圈。你只要按十字键，光圈就会根据 W3C 的空间导航算法，自动寻找上/下/左/右最近的元素跳跃过去；按下 A 键就是点击。
在这个模式下，你完全脱离了“鼠标指针”的概念，一切交互都在网格中进行。

作为一个野生开发者，我根本没系统学过操作系统（OS）的底层知识。目前项目的第一阶段（手柄底层捕获、摇杆平滑曲线、按键切层引擎、GUI 调试面板）我算是死磕下来了。
但是，那个看起来最惊艳的 Vim 空间导航模式，差点把我逼疯，现在 BUG 多到爆炸：

Windows 的 UIA 树简直是个迷宫：有些程序的层级深得离谱，扫描一次直接卡死；多线程下焦点刷新偶尔还会崩溃。
浏览器的无障碍接口巨坑：Chrome 默认是关闭 UIA 节点暴露的，经常抓不到网页元素（目前只能建议大家用 Edge，或者给 Chrome 强制加启动参数）。
不仅如此，我手头只有我自己的一台电脑和一个手柄。Xbox、PS5、甚至各种第三方手柄在不同蓝牙驱动和系统环境下的表现，我一个人根本测不过来！
